# Agent Development Guide

このファイルは、このリポジトリで今後開発するエージェント向けの作業ルールです。作業前に読み、実装・アセット作成・UI調整・記録更新をこの方針に沿って進めます。

## プロジェクトの前提

- プロジェクト名は `Chrono Reverse`。
- 目的は、時間進行と Active Response を持つ 1vs1 カードバトルを、オリジナルIP・オリジナルカードで実装すること。
- 主な実装領域は `core/`、`components/`、`pages/`、`server/`、`native/`、`public/`、`docs/`、`tests/`。
- 仕様確認では `README.md`、`teppen.md`、`UI_要件定義_改善版.md`、`docs/`、`画像生成.md`、`log.md` を優先して読む。
- カードデータは `public/新カードCore - カードデータのスプレッドシート化.csv` と `core/cards.ts`、効果実装は `core/newCardEffects.ts`、`core/cardSpecificEffects.ts`、`core/effects.ts` を確認する。

## 基本の開発手順

1. 依頼内容に関係する仕様・既存コード・既存アセットを先に確認する。
2. `git status --short` で作業前の差分を確認し、ユーザーや既存作業の変更を勝手に戻さない。
3. 変更は依頼範囲に絞る。無関係なリファクタリングやファイル整理は行わない。
4. 実装後は影響範囲に応じて `npm run type-check`、`npm run test`、`npm run build`、または対象ディレクトリの検証コマンドを実行する。
5. UI変更では、PC・スマホ横画面・スマホ縦画面ロック表示・React Native側への影響を確認する。
6. 意味のある変更をしたら、必ず `log.md` に実装内容、変更ファイル、検証結果、残課題を追記する。

## アセット作成方針

- カードイラスト、ヒーロー画像、背景、タイトル画像、UI用ビジュアルなどのラスター画像は Imagegen を利用する。
- 生成前に、アセットの用途を明確にする。
  - 例: カード画像、ヒーロー立ち絵、タイトル背景、デッキ画面背景、バトル画面装飾、UIモック。
- 生成前に `画像生成.md`、対象カードの効果・色・名前・世界観、既存画像のトーンを確認する。
- Imagegen のプロンプトには、用途、構図、被写体、背景、スタイル、比率、禁止事項を入れる。
- カード画像では、原則として画像内に文字を入れない。カード名や効果文はUI側で表示する。
- プロジェクトで使う生成画像は、最終的に `public/images/` 配下などリポジトリ内へ保存する。生成ツールの一時保存先だけに置いたまま参照しない。
- 既存アセットを置き換える場合は、ユーザーから明確に置換依頼がある時だけ上書きする。通常は `-v2` などの別名で保存して比較可能にする。
- 生成したアセットの最終パス、生成意図、主要プロンプト、UI反映箇所を `log.md` に記録する。

## アセットをもとにしたUI調整

アセットを作っただけで完了にしない。作成したアセットを実際の画面に組み込み、見え方に合わせてUIを調整する。

- 画像の比率、トリミング、余白、明度、コントラストを表示先コンポーネントで確認する。
- カード画像はカード枠、コスト表示、攻撃力、HP、効果文、レアリティ表示と干渉しないように配置する。
- ヒーロー画像・3Dモデルはホーム、デッキ選択、バトル画面で見切れや重なりが起きないように調整する。
- 背景画像は文字や操作ボタンの可読性を落とさないよう、必要に応じてオーバーレイや局所的な暗幕を使う。
- UIの主役はゲーム操作であり、装飾が操作性を邪魔しないようにする。
- 画面内テキスト・ボタン・カード・レーン・矢印・モーダルが重ならないことを確認する。
- 画像読み込み失敗時のフォールバックも確認する。

## UI実装方針

- 新しい画面を作る場合、説明用ランディングページではなく、すぐ使えるゲーム画面・管理画面・選択画面として実装する。
- 既存のコンポーネント構成を優先する。
  - Web: `components/AppShell.tsx`、`components/ScreenRouter.tsx`、`components/screens/`、`pages/`
  - Battle: `components/GameBoard.tsx`、`components/OnlineGameBoard.tsx`、`pages/battle.tsx`
  - Native: `native/src/app/`、`native/src/screens/`、`native/src/components/`
- カード、レーン、ボタン、ゲージなど固定フォーマットのUIは、安定した寸法・比率・レスポンシブ制約を持たせる。
- フォントサイズをビューポート幅だけで極端に変化させない。
- バトル画面では、タッチしやすさ、レーンの判別、カードの掴みやすさ、攻撃矢印の視認性を優先する。
- React Native側に同等画面がある場合、Webだけで完了せず、必要に応じて `native/` の設計差分も記録する。

## ゲームロジック実装方針

- ゲームルールは `core/` を正とする。UI側にゲーム判定を重複実装しない。
- Active Response、MP、攻撃タイマー、墓地、EXポケット、ヒーローアーツ、カード対象選択は既存の型とフローに合わせる。
- 新しいカード効果は、先に対象選択・イベント・状態変更の責務を確認してから実装する。
- 効果追加時は、最低限 `tests/core/` に対象効果の回帰テストを追加または更新する。
- 未実装効果は `docs/まだ未実装.md` と `log.md` に追跡できる形で残す。

## サーバー・オンライン実装方針

- WebSocketプロトコルは `core/protocol.ts` と `server/` を確認して変更する。
- オンライン対戦では、ゲーム開始時刻、タイマー、Active Response、入力順序、同期ずれを優先的に検証する。
- Renderや許可オリジンの変更は `server/allowedOrigins.ts`、`render.yaml`、`tests/server/allowedOrigins.test.ts` を確認する。

## ログ更新ルール

`log.md` はプロジェクトの実装記録として維持する。

- 変更日、目的、変更ファイル、実装内容、検証結果、未解決事項を追記する。
- アセットを生成した場合は、プロンプト要約、保存先、UI反映先を記録する。
- 仕様変更や未実装範囲の発見も記録する。
- コミット前後に関係なく、作業単位で追えるように書く。
